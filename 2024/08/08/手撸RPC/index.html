第一章 RPC 整体架构

RPC 框架介绍

RPC 全称 Remote Procedure Call，意为远程过程调用，能让开发人员像调用本地服务一样调用远程服务，是较为常用的中间件框架。一个完整的 RPC 一般可以分为服务消费者、服务提供者、注册中心、监控中心四个部分。其中服务提供者、服务消费者和注册中心。

RPC 工作流程如上图所示。

在一次远程调用过程中，服务提供者启动后，会将其提供的服务列表信息注册到注册中心。当服务消费者启动后，会向注册中心订阅服务列表。当注册中心的服务列表发生变化时，注册中心也会向服务消费者主动推送服务列表。服务消费者会从订阅的服务列表中按照一定的规则选择一个地址，将服务提供者需要的数据通过网络发送出去。服务消费者会通过动态代理的方式远程调用服务提供者提供的服务，服务消费者调用需要的接口名称、方法名称、方法参数类型列表、方法参数列表等信息序列化成二进制字节流传输给服务提供者。服务提供者接收到数据后，将数据反序列化，根据得到的数据调用对应的服务再将调用结果序列化返回给消费者。消费者反序列化数据，得到调用结果。

从上述工作流程可以知道，一个 RPC 框架的核心就是服务的注册发现、网络通信协议、序列化、调用方式、线程模型、动态代理、负载均衡等。

服务注册发现

在 RPC 框架中，通过注册中心完成服务的注册与发现。服务提供者将其提供的服务列表信息注册到注册中心，下线时从注册中心将自生移除列表。服务消费者向注册中心订阅服务列表，然后通过负载均衡算法选择其中一个服务节点调用。同时为了防止出现残留无效服务列表的问题，注册中心需要采用主动检测和心跳检测的机制，这样当服务提供者或消费者发生变化时，无需重启任何服务即可快速实现注册与发现。

调用方式

有同步调用、异步调用、回调、单向调用。

同步调用就是消费者发起调用后，线程会一直阻塞，直到提供则返回结果或者超时异常。

异步调用就是消费者发起调用后，线程不阻塞，而是获取 RPC 返回的 Future 对象，RPC 调用的结果会被服务提供者缓存起来，消费者根据情况决定获取数据，消费者获取结果的过程是阻塞的。

回调就是消费者发起调用之后，会将回调接口对象发送给 RPC 框架。当框架收到结果或者超时异常，会执行 Callback 回调。这种方式一般要实现处理成功和异常的方法（success（Object result）、fail（Exceotion e））。

单向调用就是发起调用后直接返回，忽略返回结果。

负载均衡

常见的负载均衡算法有轮询、加权轮询、最少连接数、一致性 Hash。

服务提供者注解设计

一个服务提供者注册到数据中心的元数据应该包含一下几部分：

  • 服务名称：一般是提供的服务的完整类名。
  • 服务版本号：区分不同版本所提供的服务。
  • 服务地址：服务提供者的网络地址（IP 或者域名）。
  • 服务端口号：服务提供者发布的服务所监听的端口号。
  • 服务分组：实现服务隔离的效果。

一个应用程序启动的时候只需要监听一个端口就行，不需要为应用中每个 RPC 服务监听一个端口，故服务地址和服务端口号只需要在项目配置中完成即可。

服务消费者设计

服务消费者比起服务提供者复杂的多，在 RPC 框架中同步调用、异步调用、回调、单向调用等都需要在服务消费者这一侧实现。

根据服务消费者的工作流程发现，至少需要以下信息：

  • 注册中心地址。
  • 注册中心类型，一个完善的 RPC 框架需要可以支持多种注册中心。
  • 负载均衡类型。
  • 版本号。
  • 服务分组。
  • 序列化类型。因为需要在网络中传输数据，需要序列化。
  • 超时时间。
  • 是否单向调用。
  • 是否异步调用。
  • 代理方式。

@RpcService 这个注解是标注到 RPC 服务实现类上，如果某个接口的实现类被标注了 @RpcService 注解，则这个接口和实现类会被发布为 RPC 服务，对外提供远程调用服务。

@RpcReference 这个注解是标注到类的字段，字段的引用就是标注了@RpcService 注解的实现类对应接口的实例。如果一个类的某个字段被标注，则这个类可以通过被标注的字段调用使用@RpcService 注解标注的远程服务。

注解扫描器的设计与实现

在 RPC 框架中，最核心的组件就是服务提供者和服务消费者。服务提供者和服务消费者要做到支持 Java 远程调用、整合 Spring 的 xml 配置方式调用、整合 Spring 注解调用、整合 Spring Boot 调用等等。如果要支持 Java 原生方式调用，就不能直接依赖 Spring 的方式，需要自行设计一套扫描@RpcService 和@RpcReference 注解的逻辑。

扫描器需求分析

  • 需要扫描指定包下的所有类，无论是当前包目录还是所引用的 Jar 包。
  • 扫描的文件只有.class 文件，只有.class 文件才是编译后的类文件。
  • 需要返回完整类名的集合。

可以先定义一个通用扫描器，获得当前包下全部类的类名集合 classNameList。再分别实现 @RpcService 和 @RpcReference 扫描器，对得到的 classNameList 做不同的处理。

@RpcReference 扫描的定义和@RpcService 大致相同，区别主要是@RpcReference 标注的是字段，反射找到类之后，再通过反射获得类中的全部字段。

第二章 服务提供者

在前面，我们已经完成了注解扫描器，知道了谁是服务消费者，谁是服务提供者。接下来就是完成服务提供者的具体实现。

服务提供者需要向注册中心发送消息，也需要响应服务消费者的请求并返回结果，并在启动时监听 IP 和端口号。

这个 RPC 框架的网络通信，即消息收发，依赖 netty 中的 Handler 来实现，使用的是 SimpleChannelInboundHandler 来完成消息的收发。实现就是创建一个 RpcProviderHandler 继承 netty 中的 SimpleChannelInboundHandler。
